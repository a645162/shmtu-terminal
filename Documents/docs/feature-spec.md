## 二、功能详细描述

### F01 — 启动保护（密码锁）

**优先级**：P0

**用户故事**：作为学生用户，我希望在启动应用时可以选择性设置密码保护，防止他人查看我的消费数据。

**输入**：
- 用户在设置中开启/关闭启动保护开关
- 设置保护密码（所有身份共用一个密码）
- 每次启动应用时输入密码

**输出**：
- 开启后，应用启动时弹出密码输入对话框
- 密码正确则进入应用；错误则提示重试，无重试上限
- 关闭后直接进入应用

**核心逻辑**：
1. 启动保护默认关闭
2. 密码以哈希形式存储在全局配置文件（TOML）中
3. 密码校验在 UI 层完成，不涉及业务逻辑
4. 忘记密码只能通过删除配置文件重置

**对标说明**：
- Avalonia：启动时检查全局配置，弹出模态密码窗口
- Tauri：启动时检查全局配置，弹出模态密码对话框

### F02 — 用户认证体系（身份/账号管理）

**优先级**：P0

**用户故事**：作为用户，我可能拥有多个身份（例如本人和家人的校园卡），每个身份下可能有多个学号（本科/硕士/博士不同账号）。我需要灵活管理这些身份和账号。

**输入**：
- 身份信息：名称、是否启用、是否允许同步更新
- 账号信息：账号名、学号、密码、是否启用、过期时间、最后更新时间

**输出**：
- 身份列表展示（主列表）
- 选中身份后展示其下的账号列表（子列表）
- 支持 CRUD 操作

**核心逻辑**：

#### 身份（Identity）模型
- 一个身份代表一个"人"
- 属性：名称、启用状态、是否允许同步、生日（可选）
- 每个身份拥有独立的合并数据库

#### 账号（Account）模型
- 一个账号对应一个学号/校园卡
- 属性：账号名、学号（12位）、密码、启用状态、过期时间、最后更新时间
- 每个账号拥有独立的原始账单数据库（API 获取，只读）

#### 身份选择流程
1. 应用启动后弹出身份选择窗口
2. 显示所有已启用的身份
3. 用户选择一个身份进入主界面
4. 可勾选"记住默认选择"，下次自动进入
5. 如仅有一个身份且已记住默认，则跳过选择

#### 数据隔离
- 每个账号的原始数据表：存储从 API 获取的原始账单，**只读不允许修改**
- 每个身份的合并数据表：名下所有账号的新条目自动同步到此
- 账号密码存储到加密数据库（SQLite 加密）

**对标说明**：
- Avalonia：独立窗口，左侧身份列表 + 右侧账号列表（两级列表或两个列表）
- Tauri：页面或模态对话框，同样两级列表布局

### F03 — CAS 统一认证登录

**优先级**：P0

**用户故事**：作为学生，我需要通过学校 CAS 系统登录后才能查询一卡通账单。

**输入**：
- 学号（用户名）
- 密码
- 验证码（见 F05）

**输出**：
- 登录成功：获取 ePay 会话 cookies
- 登录失败：返回错误类型（密码错误/验证码错误/其他错误）

**核心逻辑**：
1. 访问 ePay 页面，检测是否需要登录（302 = 需要，200 = 已登录）
2. 如需登录，获取 CAS 登录页面，提取 `execution` 字段
3. 获取验证码图片
4. 通过验证码识别器获取验证码（手动/远程/本地，见 F05）
5. 提交登录表单（username, password, validateCode, execution, _eventId=submit）
6. 处理 CAS 重定向链（最多10次）
7. 登录成功后保存 cookies（加密存储），下次可尝试恢复会话
8. 仅在需要同步数据时才触发登录流程，非启动即登录

**对标说明**：
- Avalonia：调用 `shmtu-dotnet-lib` 的 `EpayAuth`
- Tauri：调用 `shmtu-cas-rs` 的 `EpayAuth`
- 两端登录流程完全对等，错误类型定义一致

### F04 — 一卡通账单查询与展示

**输出**：
- 账单列表（表格形式）
- 每条记录包含：日期时间、交易名称、交易号、对方账户、金额、支付方式、状态

**核心逻辑**：

### F05 — 验证码识别

**优先级**：P0

**用户故事**：作为学生，CAS 登录需要输入验证码。我希望能选择手动输入、远程 OCR 自动识别或本地 ONNX 模型识别。

**输入**：
- 验证码图片字节流（从 CAS 获取）
- 识别模式配置（手动/远程OCR/本地ONNX）

**输出**：
- 识别结果字符串（数学表达式或最终答案）

**核心逻辑**：

#### 三种识别模式

1. **手动输入**（默认）
   - 显示验证码图片
   - 用户手动输入
   - 适用于所有场景，最可靠

2. **远程 OCR 识别**
   - 连接 TCP 服务器
   - 发送原始图片字节 + `<END>` 标记
   - 接收 UTF-8 响应（数学表达式，如 `3+5=8`）
   - 需配置 OCR 服务器地址和端口
   - 支持自动重试（最多配置次数，间隔1秒）

3. **本地 ONNX 识别**
   - 加载三个 ONNX 模型：等号分类器、运算符分类器、数字分类器
   - 图像预处理：二值化 → 裁剪等号区域 → 分类等号类型 → 分割数字/运算符 → ResNet 推理
   - 模型文件可自动从 Gitee 下载
   - 首次使用需下载模型（约几MB）

#### 验证码处理流程
1. 从 `https://cas.shmtu.edu.cn/cas/captcha` 获取验证码
2. 根据 ICaptchaResolver/CaptchaResolver 配置选择识别方式
3. 如果返回表达式类型（如 `3+5`），计算最终结果
4. 提交结果作为 validateCode

#### 验证码测试窗口
- 独立窗口/页面
- 获取当前验证码图片
- 可选择不同识别模式测试
- 显示识别结果和耗时
- 支持多次连续测试

**对标说明**：
- Avalonia：`ICaptchaResolver` 接口 + `CasOcr`（本地ONNX）+ `RemoteOcrCaptchaResolver`（远程）
- Tauri：`CaptchaResolver` trait + `OcrCaptchaResolver`（远程）+ Rust ONNX 库
- 两端识别模式和行为完全对等

### F06 — 账单增量同步

**优先级**：P1

**用户故事**：作为学生，我希望能快速同步最新账单，而不是每次都全量拉取。

**输入**：
- 选中账号或身份
- 同步选项：起始页、最大页数、账单类型、提前停止阈值

**输出**：
- 同步结果：新增条数、获取页数、是否提前停止、新增账单列表

**核心逻辑**：

#### 账号级增量同步
1. 登录 CAS → 进入 ePay
2. 从第1页开始逐页获取账单
3. 每页解析后，检查交易号是否已存在于账号原始数据表
4. 新条目写入账号原始数据表
5. 连续 N 条（默认5条）已知记录后提前停止
6. 到达最后一页也停止

#### 身份级同步
1. 遍历身份下所有启用账号
2. 对每个账号执行增量同步
3. 每个账号的新条目自动追加到身份合并数据表

#### 全量更新
- 账号级别：从第1页开始，拉取所有页，不提前停止
- 身份级别：对所有账号执行全量更新
- 全量更新后，操作记录清空（见 F09）

#### 同步状态反馈
- 显示同步进度（当前页/总页数）
- 显示新增条数
- 同步完成后提示

**对标说明**：
- Avalonia：`BillSync.IncrementalSyncAsync` + `IBillStore`
- Tauri：`incremental_sync` + `BillStore` trait
- 两端同步逻辑完全对等

### F07 — 账单数据导出

**优先级**：P1

**用户故事**：作为学生，我希望能将账单数据导出为多种格式，方便在记账软件中使用。

**输入**：
- 导出范围：账号原始数据 / 身份合并数据 / 自定义筛选
- 导出格式：CSV / JSON / 钱迹格式
- 文件保存路径

**输出**：
- 对应格式的文件

**核心逻辑**：

#### CSV 导出
- 中文表头：日期时间、交易名称、交易号、对方账户、金额、支付方式、状态
- UTF-8 BOM 编码（Excel 兼容）
- 与 Rust 库的 `CsvExporter` 输出格式对齐

#### JSON 导出
- 每条记录包含所有 BillItem 字段
- 数组格式，支持重新导入

#### 钱迹格式
- 遵循钱迹 App 的导入格式规范
- 类型映射：充值 → 收入，消费 → 支出
- 分类映射参考 F08 分类规则

**对标说明**：
- Avalonia：扩展 `shmtu-dotnet-lib` 的 export 模块（JsonFile、Qianji 当前为 stub）
- Tauri：扩展 `shmtu-cas-rs` 的 `parser::export` 模块
- 两端导出格式完全一致，确保互操作

### F08 — 账单统计分析与图表

**优先级**：P1

**用户故事**：作为学生，我希望看到消费趋势、分类占比等统计图表，了解自己的消费习惯。

**输入**：
- 选中身份的合并数据
- 时间范围：日/周/月/自定义
- 分类规则配置（TOML 格式）

**输出**：
- 统计图表：消费趋势折线图、分类饼图、食堂消费占比、日均消费等
- 统计摘要：总消费、总收入、净支出、日均消费

**核心逻辑**：

#### 分类规则（TOML 格式）

分类规则定义在配置文件中，支持 GitHub 云更新。参考已有 JSON 格式，迁移为 TOML：

**按消费类型分类**（type）：

```toml
[type]
[type.deposit]
name = "充值"
match_names = ["中行云充值", "微信充值"]

[type.electricity]
name = "电费"
match_names = ["电费缴费"]

[type.bath]
name = "洗澡"
match_targets = ["淋浴", "热水"]

[type.hot_water]
name = "热水"
match_names = ["水控转账"]

[type.cake]
name = "蛋糕"
match_targets = ["北区西点房"]

[type.canteen]
name = "食堂"
match_targets = ["食堂", "餐厅"]
```

**按位置映射**（position）：

```toml
[position]
field = "target"

[position.keywords."A食堂1楼大餐厅"]
building = "海馨楼"
room = "海馨第1食堂"

[position.keywords."A食堂2楼大餐厅"]

[position.keywords."B食堂1楼"]

room = "海琴1楼"

[position.keywords."C1大餐厅"]
building = "海联楼"
room = "海联1楼"
```

**按用餐时段分类**（schedule）：

```toml
[[schedule]]
[schedule.valid_date]
start_date = "2019.9.1"
end_date = "now"

[schedule.timetable.breakfast]
name = "早餐"
start_time = "6:30"
end_time = "8:30"

[schedule.timetable.lunch]
name = "午餐"
start_time = "10:45"
end_time = "12:30"

[schedule.timetable.dinner]
name = "晚餐"
start_time = "16:30"
end_time = "18:15"

[schedule.timetable.midnight_snack]
name = "夜宵"
start_time = "18:15"
end_time = "21:00"
```

#### 统计维度
1. **消费趋势**：按日/周/月的消费金额折线图
2. **分类占比**：按消费类型的饼图/环形图
3. **食堂分析**：各食堂消费频次和金额对比
4. **用餐时段**：早/午/晚/夜宵消费分布
5. **充值记录**：充值频率和金额趋势
6. **月度对比**：本月 vs 上月消费对比

**对标说明**：
- Avalonia：使用 ScottPlot 或 LiveCharts2 绘制图表
- Tauri：使用 Fluent UI Chart 组件
- 分类规则逻辑应抽取到公共库中，确保两端一致

### F09 — 数据导入/导出与快照

**优先级**：P1

**用户故事**：作为学生，我希望备份和恢复我的账单数据，也能在身份合并数据库上做增删操作并保留操作记录。

**输入**：
- 导入文件路径（JSON 格式）
- 快照创建/恢复操作
- 合并数据的增删操作

**输出**：
- 导入成功/失败结果
- 快照文件
- 操作记录列表

**核心逻辑**：

#### 数据导入导出
- 导出：将身份合并数据库导出为 JSON 文件
- 导入：从 JSON 文件导入数据到指定身份
- 支持选择性导入（仅导入指定时间范围）

#### 数据快照
- 创建当前数据库的完整快照
- 快照存储在独立目录中，带时间戳命名
- 可从快照恢复数据

#### 合并数据操作
- 身份数据库允许手动增删合并操作
- 每次操作记录：操作类型（添加/删除/合并）、涉及记录、操作时间
- 全量更新后操作记录清空
- 手动删除的记录在全量更新后会重新出现（因原始数据不变）

#### 操作记录模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | i64 | 主键 |
| operation_type | string | add / delete / merge |
| record_numbers | string | 涉及的交易号列表（JSON数组） |
| operation_time | datetime | 操作时间 |
| description | string | 操作描述 |

**对标说明**：
- Avalonia：独立窗口
- Tauri：页面或模态对话框

## 三、功能对标矩阵

| 功能 | Avalonia 实现 | Tauri 实现 | 共享逻辑 |
|------|--------------|------------|----------|
| CAS认证 | shmtu-dotnet-lib | shmtu-cas-rs | 登录流程对等 |
| 验证码-手动 | ICaptchaResolver | ManualCaptchaResolver | 行为一致 |
| 验证码-远程 | RemoteOcrCaptchaResolver | OcrCaptchaResolver | TCP协议一致 |
| 验证码-本地 | shmtu-ocr-onnx-lib | shmtu-ocr (Rust) | 模型一致，流程对等 |
| 账单模型 | BillItemInfo | BillItem | 字段对等 |
| 账单解析 | BillHtmlParser | parse_bill_page | 输出对等 |
| 增量同步 | BillSync | incremental_sync | 逻辑对等 |
| 数据存储 | SqlSugar + SQLite | Tauri state + SQLite | 数据格式一致 |
| 分类规则 | 公共库 | 公共库 | TOML规则一致 |
| 导出 | 扩展export模块 | 扩展export模块 | 格式一致 |

## 四、已知差异与注意事项

1. **BillType tab_no 映射不一致**：C# 端 `NotPaid=2, Success=3`，Rust 端 `Success=2, NotPaid=3`。需要在应用层统一映射。
2. **WechatAuth 实现差异**：C# 端为空壳，Rust 端已实现。热水功能暂不开发，但需注意未来对齐。
3. **验证码 ONNX 差异**：Rust 端增加了 `revert_color()` 步骤，C# 端没有，可能导致识别结果差异。
